# Руководство по локализации казино-статей

Этот документ фиксирует итоговый, практически выверенный подход к подготовке материалов `casino_articles` и полных мультиязычных cluster JSON для импорта. Он должен быть достаточным для любого следующего агента без привязки к временным рабочим папкам, промежуточным экспериментам или названиям локальных батчей.

Если старые документы расходятся с этим текстом, ориентироваться нужно на **этот** файл.

---

## 0. Самое первое правило — не затирать ручные правки

Перед любой работой с cluster JSON нужно сначала проверить, вносились ли в файл уже ручные редакторские, SEO- или структурные изменения. Такие правки нельзя перезатирать автоматической пересборкой, переводом из старого шаблона или массовой синхронизацией без явного указания пользователя.

В самом начале работы в рабочей папке обязательно создаётся backup исходного `*-full.json`. Этот backup нужен не «на всякий случай», а как рабочая точка сравнения: по нему проверяют, какие изменения уже были сделаны вручную до старта текущей задачи.

Если английская, французская, немецкая, испанская или любая другая локаль уже была отредактирована человеком, эти изменения считаются важным источником правды. Остальные локали нужно приводить к актуальному отредактированному виду, а не откатывать весь кластер к более старому scaffold или старой версии переводов.

---

## 1. Главный принцип

Для каждой статьи есть две независимые, но одинаково важные сущности:

1. **Источник правды по тексту и смыслу**
2. **Рабочая техническая структура итогового cluster-файла**

Ошибка возникает тогда, когда их путают.

### 1.1. Источник правды по тексту

Это английский текст **именно этой статьи**, согласованный с экспортом записи, архивом исходника и при необходимости с живым донором.

Из этого источника берутся:

- факты;
- порядок смысловых блоков;
- содержание абзацев;
- мета `title` / `description`;
- строки таблиц;
- FAQ;
- смысл переводов на все остальные локали.

### 1.2. Техническая структура итогового файла

Это **тот HTML-формат и тот cluster JSON-формат, который реально корректно импортируется и рендерится на целевом сайте**.

Из этого источника берутся:

- схема `seo_cluster_v1`;
- `mode: full`;
- массив `locales`;
- рабочий HTML-каркас `content`;
- локальные пути к картинкам;
- рабочие таблицы;
- правильная форма блока для фронтенда и импорта.

### 1.3. Практический вывод

Финальный файл нельзя собирать ни:

- только из донора/архивного HTML без учёта рабочего формата импорта;
- ни только из старого cluster-шаблона без сверки с каноническим английским текстом.

Правильный результат получается так:

- **смысл и факты** берутся из канонического EN;
- **рабочая структура для импорта** берётся из корректного cluster/scaffold;
- затем все локали переводятся **по этому EN**, но уже **внутри той же рабочей HTML-структуры**.

### 1.4. Безусловное правило полного цикла

Для **каждого** кластера без исключений обязателен **полный цикл работ**.

Нельзя делать сокращённую версию процесса даже если:

- в старом cluster-файле уже есть переводы;
- локали «в целом выглядят нормально»;
- нужно ускорить сдачу;
- кажется, что достаточно подправить только EN;
- хочется просто переиспользовать non-EN из предыдущей сборки.

Обязательный цикл всегда один и тот же:

1. сопоставить статью с правильной EN-версией и брендом;
2. сверить локальный EN с донором;
3. если EN урезан, добрать материал с донора;
4. сделать полноценный EN-рерайт под проектный scaffold;
5. только после этого заново перевести **все** non-EN локали именно от этого нового EN;
6. провести орфографическую, стилистическую и редакторскую нормализацию **каждой** локали;
7. нормализовать `title`, `description` и остальную meta;
8. только после всех шагов собирать финальный `full` cluster JSON.

Любая схема вида:

- «пересобрали только EN, а старые переводы оставили как есть»;
- «слегка подчистили существующие локали вместо нового перевода»;
- «взяли локали из 06 и просто поменяли пару строк»;
- «сначала собрали cluster, а meta и локали поправим потом»

считается **ошибкой процесса** и **недопустимым результатом**.

### 1.5. Если в рабочей папке нет готового примера

Отсутствие рядом готового cluster-примера, source-log или notes по конкретному бренду **не меняет процесс**.

Следующий агент должен считать, что:

- **этого гайда и `CASINO_ARTICLE_CONTENT_GUIDE.md` достаточно**, чтобы собрать правильный результат с нуля;
- пример из соседней папки может помогать как ориентир, но **не является обязательным условием** для начала работы;
- если примера нет, нужно опираться на:
  1. канонический EN и подтверждённые источники по бренду;
  2. рабочий `seo_cluster_v1` scaffold;
  3. правила по локалям, USD, картинкам и SEO из этих гайдов.

Иными словами: если пользователю потом сказать «читай гайды и делай», этого должно быть достаточно даже без локального образца в текущем батче.

## 2. Каким должен быть итоговый артефакт

Итогом работы должен быть **один полный JSON-файл** на одну запись `casino_articles`, например:

- `seo-casino_articles-<entity_id>-full.json`

Ожидаемая форма:

- `schema: seo_cluster_v1`
- `entity: casino_articles`
- `entity_id: <id>`
- `mode: full`
- `locales: [...]`

Каждый объект `locales[]` должен содержать:

- `lang_id`
- `lang_url`
- `url`
- `name`
- `title`
- `description`
- `content`
- `status`
- `source`

Обычно:

- для EN `source = "main"`
- для переводов `source = "content_i18n"`

---

## 3. Что считается правильным результатом

Правильный итоговый файл:

- импортируется без поломки структуры;
- показывает **рабочие картинки**, а не пустые блоки;
- сохраняет **рабочие таблицы**;
- использует **локальные пути** к изображениям вроде `/images/casinos/...`;
- не содержит внешнего сырого donor HTML, который ломает фронтенд;
- не содержит промокодных UI-блоков;
- не содержит валют `INR`, `BDT`, `EUR`, `RUB` и т.п. в пользовательской копии, если для проекта принят `USD`;
- даёт по каждой локали полноценный перевод этой же статьи, а не шаблонный текст другого бренда.

---

## 4. Источники правды и приоритет

При конфликте использовать такой порядок:

1. Согласованный английский исходник **этой** статьи.
2. Экспорт записи / рабочий JSON статьи в репозитории.
3. Архив исходников, если рабочая копия уже испорчена.
4. Живой донор, если нужно восстановить факты или проверить смысл.

Важно:

- лицензию;
- оператора;
- юрисдикцию;
- формулировки регулятора;
- название компании;
- номера лицензий

нельзя подменять ради шаблона, удобства перевода или унификации.

---

## 5. Жёсткие правила для английской версии

Английский в итоговом cluster-файле не является местом для нового рерайта. Это **канонический EN**, приведённый к проектным требованиям.

### 5.0. Если локальный EN урезан относительно донора

Если рабочий английский JSON:

- заметно короче донора;
- потерял важные смысловые секции;
- содержит только сокращённый пересказ вместо полной статьи;
- не покрывает таблицы, FAQ, app/install/login/bonus блоки, которые есть у донора и нужны для полноценного cluster-результата,

то **нельзя** просто переводить эту урезанную версию дальше.

В таком случае нужно:

1. взять живой донор или подтверждённый исходный экспорт этой же статьи;
2. добрать недостающий материал;
3. выполнить **новый английский рерайт** на основе донора;
4. сделать этот EN текст **существенно уникальным** по формулировкам, ориентируясь на уникальность порядка `90%+`, но без потери фактов;
5. только после этого использовать такой EN как основу для финального cluster и всех локалей.

Важно:

- это не означает «написать новую статью с нуля»;
- это означает **восстановить полноту статьи от донора**, а затем **пересобрать её в уникальных формулировках** внутри рабочего HTML-скелета проекта;
- факты бренда, лицензии, оператора и структура нужных смысловых блоков при этом сохраняются.

### 5.1. Что в EN разрешено

Если исходный EN уже полный и качественный, разрешены только такие смысловые правки:

1. **Нормализация валюты к USD**
2. **Удаление промокодов и промокодного UI**

Дополнительно допустима минимальная редактура рядом с суммой, если без неё фраза после замены валюты становится неестественной.

### 5.2. Что в EN запрещено

Нельзя:

- переписывать статью заново;
- сокращать исходник до краткого резюме;
- заменять один бренд другим;
- менять факты о лицензии, операторе, регионе;
- брать текст от другого казино;
- произвольно менять бонусные суммы, лимиты и пороги;
- вставлять новый промокодный текст;
- ломать структуру итогового HTML.

Важно: если для статьи выполнялся шаг `5.0`, то новый EN-рерайт становится обязательным каноническим основанием для **всех** дальнейших переводов. После этого нельзя возвращаться к старым non-EN локалям из предыдущего cluster-файла и нельзя считать их валидными без нового полного перевода.

---

## 6. Правила по валюте

Во всей пользовательской копии финального результата использовать `USD`, если не дано отдельное исключение.

Это относится к:

- `title`
- `description`
- заголовкам
- абзацам
- таблицам
- FAQ
- CTA-текстам

После замены валюты фраза должна остаться естественной для локали.

Нельзя:

- оставлять смешение `USD` с `INR` / `BDT` / `EUR` / `RUB` в финальном пользовательском тексте;
- менять фактологию ради валютной унификации;
- придумывать новые суммы от себя.

---

## 7. Правила по промокодам и CTA

Из финального контента должны быть удалены:

- `promocode-container`
- `input.promocode`
- `copy-btn`
- захардкоженные коды вроде `AVIATOR101`
- UI копирования кода

После удаления:

- CTA должен остаться нейтральным;
- ссылка должна вести в принятый проектом формат, обычно `href="#"`;
- нельзя придумывать новый промокод или новый промоблок.

---

## 8. Правила по структуре `content`

Это один из самых важных разделов.

### 8.1. Что нужно сохранять

В итоговом `content` должны сохраняться:

- рабочая HTML-структура для импорта;
- порядок секций;
- таблицы;
- изображения;
- порядок изображений;
- локальные `src` путей картинок;
- единый DOM-сюжет между локалями.

### 8.2. Что нельзя делать

Нельзя:

- бездумно переносить сырой donor/Gutenberg HTML в итоговый cluster, если он ломает импорт;
- менять локальные рабочие пути картинок на внешние `https://...`;
- переставлять картинки и таблицы местами без необходимости;
- переводить локали в другой HTML-каркас, чем в EN;
- держать разные структуры в разных языках.

### 8.3. Проверка результата

Если после импорта:

- пропали картинки;
- таблицы уехали;
- на фронте пустые места вместо изображений;
- структура отличается от рабочего cluster-примера,

значит был использован неправильный технический scaffold, даже если тексты сами по себе хорошие.

---

## 9. Правила для всех переводов

Каждая не-EN локаль должна быть:

- переводом **этой же статьи**;
- выполнена по каноническому EN;
- встроена в тот же рабочий HTML-каркас;
- **полноценным редакторским переводом**, а не сырым машинным прогоном;
- стилистически естественной;
- без следов машинного шаблонного текста;
- без фактов от другого бренда.

Это правило понимать буквально:

- каждая локаль переводится **как отдельный редакторский текст** под свою аудиторию;
- итог должен читаться так, будто материал написал нормальный редактор для этой локали;
- результат **не должен** выглядеть как машинный перевод, калька с английского или полуочищенный auto-translate;
- недостаточно «перевести все языки одним проходом» — **каждую** локаль нужно отдельно проверить, вычитать и довести.

Перевод должен сохранять:

- тот же `h1`;
- те же `h2` / `h3`;
- те же таблицы;
- те же FAQ;
- те же изображения;
- те же пути `/images/casinos/...`;
- те же денежные суммы в `USD`.

Перевод **не должен**:

- создавать новый сюжет;
- переносить фразы от другого казино;
- оставлять английские служебные хвосты в середине локали;
- подменять регулятора;
- писать новую статью вместо перевода.

После перевода обязательна отдельная редакторская нормализация:

- исправить машинные, дословные и неестественные обороты;
- убрать смешение английских слов с локалью там, где это не оправдано брендом или термином;
- выровнять орфографию, пунктуацию и грамматику;
- сделать `title` и `description` естественными для поисковой выдачи именно на этой локали;
- проверить, что текст читается как нормальная локальная статья, а не как сырой машинный перевод.

Практически это означает ещё и следующее:

- после первого перевода полезно **сравнить качество локалей с одним из уже сильных, живо отредактированных кластеров**, а не только с EN;
- если новая локаль звучит суше, буквальнее или «шаблоннее», чем качественный предыдущий cluster, она **не считается готовой**;
- особенно внимательно проверять локали, где модель часто оставляет кальки и англоязычные хвосты: `ru`, `ua`, `ar`, `hi`, `bn`, `pt`, `nl` и другие чувствительные языки батча;
- типичный симптом слабой локали: факты правильные, HTML целый, но meta и абзацы звучат как аккуратно очищенный auto-translate, а не как локальный редакторский текст.

Недопустимый результат:

- локаль формально переведена, но звучит как калька;
- видны машинные конструкции, неестественный порядок слов или «ломаная» грамматика;
- текст понятен по смыслу, но не проходит как естественная локальная редактура.

Если такое есть, локаль считается **не доведённой** и должна быть перепереведена или дочищена вручную.

## 10. SEO-правила

Для каждой локали:

- `title` держать примерно в пределах `<= 70` символов;
- `description` держать примерно в пределах `<= 160` символов;
- в `content` должен быть **ровно один** `<h1>`;
- у изображений должен быть осмысленный `alt`.

Мета должна:

- соответствовать бренду и теме;
- быть читаемой;
- не быть машинным мусором;
- не быть копией от другого slug.

## 12. Типовые ошибки, которые нельзя повторять

### Ошибка 1. Взять правильный текст, но неправильный HTML-каркас

Следствие:

- картинки ломаются;
- импорт даёт пустые места;
- фронтенд рендерит страницу криво.

### Ошибка 2. Взять рабочую структуру, но забыть выровнять её по каноническому EN

Следствие:

- факты расходятся с исходной статьёй;
- локали переводят не тот текст;
- появляются шаблонные остатки.

### Ошибка 3. Переводить не от EN этой статьи, а от шаблона предыдущего бренда

Следствие:

- остаются чужие лицензии;
- меняются суммы;
- локаль выглядит правдоподобно, но фактически неверна.

### Ошибка 4. Оставлять внешние изображения из донора

Следствие:

- на сайте появляются битые или пустые картинки;
- статья зависит от внешнего источника вместо локальных ассетов.

### Ошибка 5. Считать, что временные рабочие папки важнее принципа

Следствие:

- агент копирует процесс, но не понимает цель;
- следующая сборка снова уходит в неверный формат.

Папки, батчи и промежуточные шаги могут быть любыми. Важно только:

- откуда берётся **смысловой EN**;
- откуда берётся **рабочая структура импорта**;
- каким должен быть **итоговый full cluster JSON**.

### Ошибка 6. Обновить EN, но не перевести локали заново

Следствие:

- переводы больше не соответствуют новому каноническому EN;
- в локалях остаются старые смысловые акценты и прежняя meta;
- часть текста выглядит «почищенной», но процесс всё равно нарушен.

Это запрещено. Если EN поменялся содержательно, все локали переводятся заново.

## 13. Посты в блоге (`entity: blog`)

Цикл тот же, что для `casino_articles`: канонический EN по смыслу + неизменный рабочий HTML-каркас `seo_cluster_v1` + полный перевод всех локалей + редакторская нормализация.

Дополнительные правила именно для блога:

- **Имя продукта `Aviator`** — собственное имя в латинице: не переводить на «авиатор», «пилот», «Flieger», «Aviateur» и т.п. в заголовках, мета и теле; в промптах автопилота зафиксировано в `ai_prompt_templates` (см. defaults в коде).
- **Структура HTML**: тот же счётчик тегов, что и у EN (см. `translation_structure_lock`); типичные сбои модели — дублирование блоков `<li>`, «слияние» шагов инструкции с разделом советов, обрыв `</ul>` — такие фрагменты при ручной вычитке удаляют и выравнивают по EN.
- **Meta**: после перевода выровнять длину `title` / `description` под привычные SEO-лимиты, без смешения языков и без лишних калек из английского там, где в языке принят другой порядок слов.
- **Векторная БД / RAG**: если для перевода подмешиваются примеры, добавляйте чанки с явным правилом «не переводить бренд Aviator» и «сохранять число элементов списков как в источнике»; иначе NVIDIA/LLM будет повторять те же дрейфы бренда и вёрстки.
- **Импорт через SEO Monitor (полный кластер)**: материал по-прежнему проходит ту же валидацию (`translation_cluster_refresh_state` / SEO Monitor rules). Если после импорта в `content_i18n` есть **все языки из scope кластера** (как в SEO Monitor и в `translation_settings.enabled_lang_ids`), в `translation_cluster_state` выставляется **`seo_monitor_handoff`**: это пометка «ручная доводка», автопилот **не ставит** на такой кластер задачи перевода/пайплайна/meta-fix, а пары EN→локаль поднимаются в `translation_vector_items` с **`approved`** как эталон для RAG.

Импорт и проверка — как для кластеров казино-статей (схема, `mode: full`, отсутствие битой разметки).

## 14. Короткая инструкция для следующего агента

Если нужно собрать или пересобрать локализованный cluster для `casino_articles`, делай так:

1. Даже если в рабочей папке нет готового примера, считай эти гайды достаточной инструкцией и собирай кластер по общему процессу.
2. Найди канонический английский источник **именно этой статьи** и сопоставь его с правильным брендом.
3. Сверь локальный EN с донором и при необходимости добери недостающий материал.
4. Построй полноценный EN-рерайт внутри рабочего технического scaffold cluster-файла.
5. Сохрани рабочие картинки, таблицы и локальные пути к изображениям.
6. Переведи все локали **заново** по этому финальному EN, не меняя HTML-каркас.
7. Отредактируй каждую локаль до нормальной орфографии, стилистики и читаемости.
8. Нормализуй `title`, `description` и остальную meta после перевода.
9. Сравни итоговые локали с одним из уже сильных finished-cluster по уровню естественности; если новый batch звучит слабее, вернись к ручной редактуре, особенно по `ru` / `ua` и другим чувствительным локалям.
10. Для поиска фактов и визуала можно запускать субагентов параллельно, но финальная фактология и пригодность ассетов всё равно проверяются вручную по приоритету источников из этого гайда.
11. Только теперь собирай итоговый `full` cluster JSON.
12. Проверь, что итоговый файл:
   - имеет схему `seo_cluster_v1`;
   - содержит `mode: full`;
   - не содержит внешних картинок, промокодов и не-USD валют;
   - сохраняет факты бренда;
   - готов к импорту без дополнительных ручных исправлений.
